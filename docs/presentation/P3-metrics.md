> 驗收：定義五大指標並用 #14 (S7) 端到端產出實測填數字。
> **誠實原則**：本文件完成「定義 + 量測方法 + 目標值」；實測欄位一律 `〈待 #14 實測〉`，端到端在真實 AWS 跑通後回填，不編造。

## 指標總表

| 指標 | 定義 | 量測方法 | 目標值 | 實測 |
|------|------|----------|--------|------|
| 剪輯時間節省比例 | (人工剪輯耗時 − 系統端到端耗時) ÷ 人工剪輯耗時 | 同一素材：碼錶記錄「人工從看片到匯出」vs「上傳到 `ARTIFACT_READY`」 | ≥ 80% | 〈待 #14 實測〉 |
| 高光偵測準確率 | 系統 top-N 高光與人工標註的重疊度 | 人工標註 ground-truth 高光時段 → 算 Precision / Recall / F1（IoU ≥ 0.5 視為命中） | Precision ≥ 0.6 | 〈待 #14 實測〉 |
| 短片品質評分 | 成品短片的主觀品質 | 3–5 位評分者對 final.mp4 打 1–5 分（節奏、字幕正確性、切點自然度） | 平均 ≥ 3.5 / 5 | 〈待 #14 實測〉 |
| 自動化程度 | 從上傳到「可編輯草稿」所需人工操作步數 | 計算 `READY_TO_EDIT` 前的人工介入次數（理想 = 僅「建 Project + 上傳」2 步） | ≤ 2 步 | 〈待 #14 實測〉 |
| 商業效益 | 每支短片的邊際時間/成本 | 端到端 wall-clock（分鐘）+ 粗估 AWS 成本（Lambda/Transcribe/Batch/S3） | < 5 分鐘 / 支 | 〈待 #14 實測〉 |

## 量測情境（給 #14 S7 執行者）
1. **固定素材**：準備 1–3 支代表性直播片段（建議 10 / 30 / 60 分鐘各一），連同人工標註的高光時段。
2. **走完整流程**：建 Project（目標 30s）→ 上傳 → 等 `READY_TO_EDIT` → 不編輯直接 Render → 下載 `final.mp4`。
3. **記錄**：每階段時間戳（`created_at / updated_at / render started/completed`，皆 ms）、highlights.v1 JSON、final.mp4。
4. **回填**：把上表實測欄與 P5 測試紀錄、P4 截圖對齊。

## 量測腳本提示
- 端到端耗時可由 DynamoDB Project item 的時間戳直接算：`ARTIFACT_READY.updated_at − CREATED.created_at`（ms）。
- 高光準確率：把人工標註時段與 `highlights.v1.json` 的 `[start_ms, end_ms]` 做 IoU 比對。
- 成本：以 dev 一次跑的 CloudWatch/Batch 用量粗估；MVP 階段給數量級即可。
